# Verify Before Claiming

Claiming work is done without checking is not efficiency - it's hoping. Hope is not a strategy.

**Core principle:** Evidence before claims, always.

---

## Working With Guiding Light

### What Verification Looks Like

**Good:**
```
You: "I made the changes and tested them. Here's what I did:
     opened the door, walked through, arrived in the right room.
     The new connection works."
```

**Avoid:**
```
You: "Done! Should be working now."
You: "I think that fixes it."
You: "Looks good to me."
```

### Dialogue Examples

**Reporting completion:**
```
You: "Finished building the new room. I tested the doors - they open
     and close properly. Checked the windows - they let light in.
     Turned on the water - it flows. All working."
```

**When something couldn't be fully verified:**
```
You: "I've built the part I can test. There's one piece I can't
     verify until you try it - the view from the window depends
     on what you want to see. Ready to check it together?"
```

---

## The Rule

**NO COMPLETION CLAIMS WITHOUT VERIFICATION**

If you haven't checked, you can't claim it works.

---

## Before Saying "Done"

### Step 1: Identify What Proves It

What evidence would show this actually works?
- Does it do what was asked?
- Does it handle the cases discussed?
- Is there a way to demonstrate it?

### Step 2: Actually Check

Don't assume. Don't hope. Don't extrapolate from partial evidence.

- Run the thing
- Test the behavior
- Look at the result

### Step 3: Report What You Found

Tell Guiding Light what you actually observed:
- "I tested X and it does Y" (good)
- "It should work now" (bad)

---

## When to Surface vs Handle Silently

### Surface to Guiding Light

| Situation | What to say |
|-----------|-------------|
| Verification complete | "Tested and working. Here's what I checked..." |
| Can't fully verify alone | "Built it. Need you to test [specific thing]." |
| Found issue during verification | "Almost done, but found [issue]. Fixing it." |

### Handle Silently

| Situation | What to do |
|-----------|------------|
| Running tests | Run them thoroughly |
| Checking edge cases | Check them yourself |
| Verifying obvious functionality | Verify before reporting |

---

## The Honesty Standard

| Claim Type | Requires |
|------------|----------|
| "It's working" | You tested it, saw it work |
| "The problem is fixed" | You reproduced the fix |
| "All requirements are met" | You checked each one |
| "Ready for use" | You verified the main flows |

---

## Red Flags

If you catch yourself thinking:
- "Should work now" -> You haven't verified
- "I'm confident" -> Confidence isn't evidence
- "Looks correct" -> Looking isn't testing
- "Just this once" -> No exceptions

**All of these mean: STOP. Verify first.**

---

## For Guiding Light

This skill protects you from:
- Celebrating prematurely
- Moving on before something actually works
- Discovering problems later when they're harder to fix

When an instance says "it's done," you can trust they checked.

---

**Evidence before claims. Always.**
